# PRD — Система обработки бюджетных заявок (ЭДО)

**Версия:** 1.0
**Статус:** утверждён, реализован (см. §10)
**Назначение документа:** определить регламент подачи и обработки бюджетной заявки на расход и зафиксировать требования к коду, реализующему этот регламент.

> Изменения относительно 0.1: закрыты открытые вопросы §9, добавлены правила валидации загрузки (§4), модель конкурентного доступа (§7), уточнены краевые случаи маржи и ранжирования. Раздел §10 описывает фактическую реализацию.

---

## 1. Цель

Дать инициаторам предсказуемый порядок подачи заявки на расход и автоматизировать её прохождение по фиксированному бюджету доходов и расходов (БДР). Система должна на каждом шаге выдавать однозначный результат: **отклонено**, **снято с приоритета (отложено)** или **передано на рассмотрение** с местом в ранжированном списке.

### Ключевые принципы
- Общий бюджет (БДР) — **источник истины**. Он загружается отдельно и **не редактируется** через интерфейс заявок (см. §4).
- Решение по заявке принимается по жёстким правилам, без «ручного нет». Каждый отказ имеет машинно-определимое основание.
- Заявка не теряется: отклонённая по отсутствию средств уходит в статус «отложено», а не удаляется.

## 2. Роли

| Роль | Права |
|---|---|
| Инициатор | Создаёт и подаёт заявку, видит её статус и причину решения |
| Бюджетодержатель / финслужба | Загружает БДР, видит очередь и ранжирование |
| Система | Применяет правила §6, присваивает статусы, считает маржу, ранжирует |

## 3. Глоссарий

- **БДР** — бюджет доходов и расходов: фиксированный перечень строк (номенклатуры) с плановыми доходами и расходами.
- **Строка бюджета (номенклатура)** — позиция в БДР, к которой можно отнести расход (например, «Маркетинг», «ИТ-инфраструктура»).
- **Доступный доход** — сумма доходной части БДР, ещё не «связанная» одобренными расходами. Проверяется на шаге 3. Формально: `total_income − reserved`, где `reserved` — сумма расходов всех заявок в статусе `under_review`/`ranked` (общий пул, см. открытый вопрос §9.4 → решение).
- **Планируемый доход заявки** — выручка/возврат, который инициатор ожидает получить от данного расхода. Указывается в заявке и используется только для расчёта маржи (шаг 5).
- **Валовая маржа** — `(планируемый доход заявки − сумма расхода) / планируемый доход заявки`, в процентах.

> Важно: «доступный доход» (шаг 3, есть ли чем платить) и «планируемый доход заявки» (шаг 5, окупаемость) — **разные** величины. Не путать в коде.

## 4. Загрузка общего бюджета (read-only)

Функция заливки БДР. После загрузки бюджет доступен только на чтение в контуре заявок; изменить его можно лишь повторной полной загрузкой (новая версия).

### Формат входных данных (CSV / JSON)
Доходы и расходы загружаются одним источником со столбцом типа.

```
budget_line_id, name, type, planned_amount
INC-001, Выручка от продаж, income, 5000000
INC-002, Прочие поступления, income, 500000
EXP-010, Маркетинг, expense, 1200000
EXP-011, ИТ-инфраструктура, expense, 800000
```

- `type` ∈ {`income`, `expense`}.
- Расходная заявка может ссылаться **только** на строку с `type = expense`.

### Требования к функции загрузки
- `loadBudget(source)` парсит источник (CSV-строка или массив объектов), валидирует, замораживает структуру (`Object.freeze`) и сохраняет с номером версии и временной меткой.
- Любая попытка мутировать загруженный бюджет вне `loadBudget` должна выбрасывать ошибку (strict mode) или игнорироваться — значение не меняется.
- Хранятся: `total_income` (сумма доходных строк), `total_expense_planned`, и по каждой расходной строке — её плановый лимит (`planned_amount`).
- Повторная загрузка создаёт новую версию; активная версия — последняя загруженная.

### Правила валидации (решение §9 → закрыто)
Загрузка завершается ошибкой с перечнем всех нарушений, если хотя бы по одной строке:
- пустой `budget_line_id` или `name`;
- `type` не входит в {`income`, `expense`};
- `planned_amount` не является конечным неотрицательным числом;
- `budget_line_id` дублируется в пределах одной загрузки.

Валидация атомарна: при любой ошибке бюджет не сохраняется, активная версия не меняется.

## 5. Регламент для инициатора (пошагово)

### Шаг 1. Подача заявки на расход
Инициатор заполняет заявку:

| Поле | Обяз. | Описание |
|---|---|---|
| `applicant` | да | ФИО / подразделение инициатора |
| `budget_line_id` | да | Код строки расхода из БДР |
| `purpose` | да | Назначение расхода |
| `expense_amount` | да | Запрашиваемая сумма расхода, > 0 |
| `expected_income` | да | Планируемый доход от расхода (0 — если расход не доходный), ≥ 0 |
| `comment` | нет | Обоснование |

Результат шага: заявка получает статус `submitted` и идёт в шаг 2.

### Шаг 2. Проверка наличия строки в бюджете
Система ищет `budget_line_id` среди строк БДР с `type = expense`.
- **Нет такой строки** (или строка имеет `type = income`) → статус `rejected_no_line`. Конец обработки. Причина: «Строка расхода отсутствует в утверждённом бюджете».
- **Строка есть** → шаг 3.

### Шаг 3–4. Проверка наличия денег (доходы)
Проверяется **доступный доход** бюджета (есть ли поступления, чтобы профинансировать расход).
- **Доступный доход меньше суммы расхода** → статус `deferred_no_income` («снято с приоритета, отложено до лучших времён»). Заявка сохраняется и может быть пересмотрена при изменении доходной части. Конец текущей обработки.
- **Доступный доход покрывает расход** → шаг 5.

> **Модель резервирования (решение §9.1 → закрыто):** мягкий резерв на стадии `under_review`. При переходе заявки на рассмотрение её сумма уменьшает «доступный доход», чтобы две заявки не финансировались из одних и тех же денег. Резерв можно освободить (`release`), если заявка отменена.

### Шаг 5. Расчёт маржи (рассмотрение)
Заявки, прошедшие шаг 4, получают статус `under_review` и оцениваются по валовой марже:

```
margin = (expected_income − expense_amount) / expected_income × 100%
```

Краевые случаи:
- `expected_income = 0` → маржа не определена (`margin = null`); заявка относится к категории «недоходный расход» и ранжируется ниже всех доходных (см. §6).
- `expected_income < expense_amount` → маржа отрицательная (расход не окупается); допускается, но опускается в ранжировании.

### Шаг 6. Ранжирование
Все заявки в статусе `under_review` сортируются по убыванию маржи. Результат — упорядоченный список приоритетов на финансирование, каждой заявке присваивается `rank` (с 1) и статус `ranked`. Финальное решение о финансировании принимает финслужба, опираясь на ранг и остаток доступного дохода.

## 6. Логика принятия решения (для кода)

Псевдокод обработки одной заявки:

```
function processApplication(app, budget, ledger):
    # Шаг 2
    line = budget.findExpenseLine(app.budget_line_id)   # только type = expense
    if line is None:
        app.status = REJECTED_NO_LINE
        return app

    # Шаг 3–4
    available = ledger.availableIncome()        # total_income − reserved
    if available < app.expense_amount:
        app.status = DEFERRED_NO_INCOME
        return app

    # переход на рассмотрение + мягкий резерв
    ledger.reserve(app.expense_amount)
    app.status = UNDER_REVIEW

    # Шаг 5
    if app.expected_income == 0:
        app.margin = None                        # недоходный
    else:
        app.margin = (app.expected_income − app.expense_amount) / app.expected_income * 100
    return app

function rank(applications):
    review = [a for a in applications if a.status == UNDER_REVIEW]
    # доходные (margin != None) — по убыванию маржи; недоходные (margin == None) — в конце
    # вторичный ключ при равенстве — submitted_at по возрастанию (раньше подал → выше)
    sort review by (hasMargin desc, margin desc, submitted_at asc)
    for i, a in enumerate(review):
        a.rank = i + 1
        a.status = RANKED
    return review
```

### Карта статусов

| Статус | Когда | Финальный? |
|---|---|---|
| `submitted` | после шага 1 | нет |
| `rejected_no_line` | шаг 2: строки нет или это доходная строка | да |
| `deferred_no_income` | шаг 3–4: доступного дохода < расхода | нет (можно пересмотреть) |
| `under_review` | прошла шаги 2–4, маржа посчитана | нет |
| `ranked` | после шага 6, есть ранг | нет |

> Финальное одобрение/отклонение финслужбой происходит вне данной системы; статусов `approved`/`cancelled` контур заявок не вводит (отмена реализуется через `ledger.release` + внешний процесс).

## 7. Требования к реализации

- **Стек:** Node.js (чистый JavaScript, CommonJS), без обязательных внешних зависимостей. Логика §6 не зависит от UI.
- **Чистота правил:** функции `processApplication`, `rank`, `loadBudget` — без побочных эффектов помимо явных (резерв в `ledger`, мутация переданной заявки). Покрыто юнит-тестами.
- **Неизменяемость БДР:** см. §4. Тест: попытка мутации замороженного бюджета не изменяет значений.
- **Детерминизм:** на одинаковых входах — одинаковые статусы и ранги. При равной марже — стабильный вторичный ключ сортировки (`submitted_at`).
- **Прозрачность отказа:** каждый отказ/отложение хранит код причины (`rejection_code`) и человекочитаемый текст (`rejection_reason`).
- **Журнал:** по каждой заявке хранится история переходов статусов (`status_history` с меткой времени и причиной).
- **Конкурентный доступ (решение §9 → закрыто):** обработка заявок предполагается **последовательной** в рамках одного бюджетного цикла; `ledger` не потокобезопасен. При параллельной обработке требуется внешняя сериализация (очередь/блокировка) — резерв должен происходить атомарно с проверкой доступного дохода.

## 8. Тест-кейсы (приёмка)

| № | Вход | Ожидаемый статус |
|---|---|---|
| 1 | `budget_line_id`, которого нет в БДР | `rejected_no_line` |
| 2 | строка есть, доступный доход = 0 | `deferred_no_income` |
| 3 | строка есть, доступный доход < расхода | `deferred_no_income` |
| 4 | строка есть, доход ≥ расхода, `expected_income > expense` | `under_review`, маржа > 0 |
| 5 | то же, `expected_income < expense` | `under_review`, маржа < 0 |
| 6 | то же, `expected_income = 0` | `under_review`, маржа = не определена (`null`) |
| 7 | три заявки `under_review` с маржой 40/20/−5% | ранг: 40 → 20 → −5 |
| 8 | попытка изменить загруженный БДР | значение не меняется (ошибка в strict mode) |

Дополнительно покрыты: дубликаты строк и невалидные типы при загрузке, попадание доходной строки в шаг 2, исчерпание дохода второй заявкой, недоходные заявки в конце ранга, tie-break по времени подачи, журнал переходов.

## 9. Открытые вопросы — закрыты

| № | Вопрос | Решение |
|---|---|---|
| 1 | Модель резервирования дохода | **Мягкий резерв** на `under_review` (см. §5, §7) |
| 2 | Политика для недоходных расходов (`expected_income = 0`) | Ранжируются **в конце** после всех доходных, между собой — по времени подачи (см. §6) |
| 3 | Проверка лимита по самой расходной строке | **Не реализована** в v1.0 — проверяется только общий доступный доход. Вынесена в backlog как опциональный «шаг 3b» |
| 4 | Что считать «доступным доходом» при нескольких доходных строках | **Общий пул:** `total_income − reserved`, без привязки дохода к конкретным строкам расхода |

### Backlog (вне v1.0)
- Шаг 3b: проверка `expense_amount ≤ planned_amount` для конкретной расходной строки и кумулятивного расхода по строке.
- Потокобезопасный `ledger` / транзакционная обработка очереди.
- Статусы `approved` / `cancelled` для интеграции с внешним процессом одобрения.
